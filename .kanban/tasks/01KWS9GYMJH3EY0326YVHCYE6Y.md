    **Bug**: `performRequest` writes a request to the child process's stdin *before* `awaitResponse(id:)` ever registers a `CheckedContinuation` for that request's id in `PendingRequestTable` (the write is synchronous; registration happens inside a separately-scheduled `withThrowingTaskGroup` child task). Nothing enforces that registration wins the race against a real round trip. `route(message:...)` called `pendingRequests.resolve(id:with:)` and ignored its `Bool` return — so if a fast response arrived before the continuation registered, it was silently discarded. The registration that followed was then orphaned: only its own sibling timeout task (a real 30s `ContinuousClock` sleep by default) could ever resume it, and if that timeout task's own wakeup was starved by the same cooperative-thread-pool pressure that lost the original race (exactly what several hundred concurrently-running tests, or TSan's much heavier per-op overhead, create), the continuation — and the whole `Task` awaiting it — hung forever with zero CPU activity. This exactly matches the task's thread-sample evidence (main thread parked in mach_msg/CFRunLoop, one dispatch worker idle in __workq_kernreturn, nothing blocked in a syscall).

    **Fix**: `PendingRequestTable` now buffers an early-arriving response (`resolveOrBuffer(id:response:)`, used only by `route()`) when nothing is registered yet, instead of discarding it; `register(id:continuation:)` checks that buffer first and delivers immediately if something is already waiting. This makes the register/resolve ordering safe regardless of which side wins the race — no more silent loss, no more orphaned continuation. `resolve(id:with:)` (used by the timeout race) is unchanged since a miss there always legitimately means "already resolved," never "not yet registered."
